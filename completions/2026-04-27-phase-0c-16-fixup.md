TASK 1 — Fix Mode A / Mode B voice form leak
═══════════════════════════════════════════════════════════════════════

CONTEXT

Lived use of v0.5.4 revealed: when an ambient panel-transition event 
fires (panelAwakened, panelEngaged, panelDismissed) and a transcript 
band would appear, the VoiceForm (120pt amber circle with rings) is 
ALSO rendering on top of the photo. This is wrong.

Per spec:
- Mode A (mic tap): voice form 120pt centered overlay + surface dim + 
  transcript prominent
- Mode B (panel transitions): NO voice form, NO overlay, NO surface dim 
  — just the discreet bottom transcript band

The voice form is leaking into Mode B somewhere in the rendering path.

DIAGNOSE

Likely causes (CC: investigate which is correct):

1. VoiceModalOverlay's visibility condition is wrong — it's showing 
   when presentationState is .ambientSpeaking when it should only show 
   when .modalSpeaking
2. VoiceForm is being rendered separately, outside VoiceModalOverlay, 
   and not gated by mode
3. The presentationState publishing is conflating modes — checking what 
   state is set when each event type fires

FIX

Whatever the cause, the constraint is:
- VoiceForm renders ONLY when presentationState == .modalSpeaking
- VoiceModalOverlay shows ONLY when presentationState == .modalSpeaking  
- Surface dim activates ONLY when presentationState == .modalSpeaking
- AmbientTranscriptBand shows ONLY when presentationState == .ambientSpeaking

Verify by inspecting VoiceModalOverlay.swift, VoiceForm.swift, RootView 
(or wherever overlays are wired), and VoiceService's presentationState 
publishing logic. Make sure the .ambientSpeaking case publishes only 
the band-relevant state and NOT anything that triggers VoiceForm 
rendering.

═══════════════════════════════════════════════════════════════════════
TASK 2 — Add visible chrome to AmbientTranscriptBand
═══════════════════════════════════════════════════════════════════════

CONTEXT

In lived use of v0.5.4, the ambient transcript appears as text floating 
on the screen with no visible backing, padding, or border. It looks 
like debug output or glitch text rather than a deliberate UI element.

Per spec, AmbientTranscriptBand should have:
- Background: hearthStatusBg (semi-transparent dark) — but this may 
  be too subtle for legibility on busy photos
- Padding: 24pt vertical, 32pt horizontal
- TranscriptText (.ambient size, 22pt) centered

DIAGNOSE AND FIX

Investigate AmbientTranscriptBand.swift. Possible issues:

1. Background isn't being applied (modifier dropped or hearthStatusBg 
   resolves to fully transparent)
2. Padding isn't being applied
3. The HStack/container structure isn't taking full width
4. Z-ordering puts the band behind something that's hiding the chrome

REVISED VISUAL TREATMENT

The hearthStatusBg backing turned out to be too subtle for legibility 
on photos in ambient mode. Strengthen the band visual treatment:

- Background: Color.black.opacity(0.75) — substantially darker than 
  hearthStatusBg, ensures legibility on bright photos
- Optional border: 1pt hearthAmberDim at 30% opacity — subtle signal 
  that this is a deliberate UI element rather than text overlay
- Padding: 24pt vertical, 32pt horizontal — keep as specced
- Full width with edge-to-edge background (no rounded corners, treat 
  as a band that spans the screen)
- Position: bottom of screen, ABOVE status bar (per spec) — verify 
  in ambient state, quick control state, full control state

The "discreet" intent stays — this isn't a modal takeover. But the 
band needs to be unmistakably present rather than apologetic.

TASK 3 — Fix transcript band position in ambient state
═══════════════════════════════════════════════════════════════════════

CONTEXT

Image observed: transcript "The house has been waiting for you." 
floating mid-photo (~60% screen height) instead of bottom-above-
status-bar.

Per spec (brief Section 11.10, Mode B positioning):
- AMBIENT STATE: band sits at bottom of screen, ABOVE the status bar 
  (status bar is at the very bottom in ambient state)
- QUICK CONTROL STATE: band at bottom of screen, separate from controls
- FULL CONTROL STATE: band in gap between cards and bottom action 
  tile row

DIAGNOSE AND FIX

The band should attach to the bottom edge of the screen with 
.padding(.bottom, HearthStatusBar.height + HearthSpacing.xs) per the 
0c.16 completion report's intent. Verify this calculation actually 
produces "above status bar" positioning in all three panel states.

If the band is rendering mid-screen, likely causes:
1. .padding(.bottom, ...) isn't being applied because the band isn't 
   inside a layout that respects bottom alignment
2. ZStack ordering puts the band in a position that doesn't anchor 
   to bottom
3. RootView's overlay attachment is centering rather than bottom-
   aligning
