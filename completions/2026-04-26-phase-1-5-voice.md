## Voice Architecture — Critical Findings

This section documents what was learned through extensive hardware testing. It changes the long-term voice strategy.

### What we tried (in order)

1. **Named Siri compact identifiers** (`com.apple.ttsbundle.siri_Arthur_en-GB_compact`, `siri_Reed_en-GB_compact`, `siri_male_en-GB_compact`, etc.) — all returned `nil`. These identifiers don't resolve via `AVSpeechSynthesisVoice(identifier:)` on either device.
2. **Premium identifier variants** (`siri_male_3_en-GB_premium`, etc.) — also all returned `nil`. The `_premium` suffix is not a valid scheme.
3. **`AVSpeechSynthesisVoice(language: "en-GB")`** — ✅ resolves correctly, returns whatever the user has set as their system default en-GB voice in Settings → Accessibility → Live Speech → Voice. This IS the right approach.

### The real problem: iOS version, not voice identifier

- **iPad Pro 2nd gen (iOS 17)**: Even with Voice 2 set as system default and downloaded, the Siri neural voice renders noticeably worse — more robotic, less natural — than on the Air.
- **iPad Air M2 (iOS 18)**: Voice 2 sounds excellent. Natural, warm, clearly neural quality.
- Same voice setting. Same identifier. Different render quality. iOS 17 vs 18 is the differentiator.

### Settings that work (iPad Air, iOS 18)
- Settings → Accessibility → Live Speech → English → Voice: Voice 2
- Rate: ~40% of slider
- Pitch: 0 (far left, minimum)
- Mapped to API values: rate `0.40`, pitchMultiplier `0.5`

### Future recommendation (for DC planning)

**AVSpeechSynthesizer is not the right long-term tool when voice quality matters across devices.**

The wall panel iPad Pro is iOS 17 hardware. Its Siri voice render quality is audibly inferior. The recommended future architecture:

> **Mac mini renders audio (pre-rendered or real-time via a high-quality TTS pipeline) and plays/streams to the iPads.**

Options to evaluate:
- Pre-render all phrases as `.m3u8`/`.caf` files on Mac mini, distribute as app bundle assets — `AVAudioPlayer` playback, zero synthesis on device
- Real-time: Mac mini runs a TTS service (e.g. macOS `say` command, OpenAI TTS, or ElevenLabs), iPad receives audio stream via local network

This is a Phase 2+ item. Phase 1.5 ships with `AVSpeechSynthesisVoice(language: "en-GB")` as a working foundation — it sounds good on the Air and acceptable on the Pro.

## Iteration Log (CC session, 2026-04-26)

This was an unusually long CC session with extensive hardware testing. Key decisions made:

| Decision | Rationale |
|----------|-----------|
| `AVSpeechSynthesisVoice(language:)` over identifier probing | Identifiers all returned nil; language: picks up system default at correct quality |
| Rate 0.40 | ~40% of iOS Settings rate slider; slower = more deliberate, JARVIS-like |
| pitchMultiplier 0.5 | Minimum value; deepens voice toward JARVIS character |
| JARVIS phrase bank | Todd's direction — replaces original Hearth-aesthetic phrases |
| Version bump per deploy | Added to process after Todd flagged confusion with stale 0.5.0 showing on device |
| Diagnostic first-tap still present | Useful for confirming voice identity during iteration; remove before tagging |

### Version/build confusion issue
Multiple builds deployed as "0.5.0 / build 50" because `Info.plist` wasn't being updated. Fixed by bumping `CFBundleShortVersionString` and `CFBundleVersion` in `Info.plist` on every deploy. CC must always do this going forward.

### Install reliability issue
Several builds reported "BUILD SUCCEEDED" but the app on device didn't update — `xcodebuild` completed without an "Installing" step, likely due to device being mid-reboot or DB lock from concurrent builds. Fix: kill stale xcodebuild processes, delete `build.db`, redeploy.
