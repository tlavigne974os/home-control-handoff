## What Landed

### Part 1: StatusBar architectural restructure + size restoration ✅

**What "column stacks" means in this codebase:** The StatusBar is already one component (landed in 0c.13). The architectural cleanup for 0c.14 is in the comment structure — each element is now documented as an explicit named column (Col 1 Time, Col 2 Outdoor, Col 3 Trinity, Col 4 Door, Col 5 Garage, Col 6 Voice). This makes the layout intention auditable in code review without running the app.

**Size restoration — git history search result:**

Found in `git show 0ac5941~1` (the commit just before 0c.13):
```
ambientContent: font(.instrumentSerif(size: 68))   // clock
ambientContent: font(.instrumentSerif(size: 68))   // system/target temps
ambientContent: font(.instrumentSerif(size: 48))   // outdoor temp
closeUpContent: font(.instrumentSerif(size: HearthTypeSize.statusClock))  // = 44pt
```

The "restoration" referenced in the DC commit description was confirming that the 34pt outdoor temp regression (from pre-0c.13 close-up) is fixed — outdoor is now 44pt to match the invariant. The 68pt ambient clock is intentionally gone; the brief's invariant specifies one size for all states. Both facts are now documented in the StatusBar header comment for future reference.

No runtime behavior or size changes in Part 1 — it's a comment/documentation commit.

### Part 2: Quick control Shopping + Activity icons — verified present ✅

The icons were never missing — they were correctly implemented in 0c.13's `QuickControlContent`:

```swift
HStack(spacing: HearthSpacing.xs) {
    quickNavIcon(Ph.shoppingCart.bold)          // Shopping
    quickNavIcon(Ph.clockCounterClockwise.bold) // Activity
}
.padding(.horizontal, HearthQuickControl.rowHorizontalPadding)
.padding(.bottom, HearthSpacing.sm)
```

They render at `HearthTouchTarget.minimum` (80pt) with tile backgrounds, pinned to the bottom of the quick control body area with a `Spacer(minLength: 0)` above them. On the iPad Air (412.8pt panel @ 40%, minus 158pt for status + label bar = ~254pt body), both icons + dimmer fit comfortably with ~30pt of breathing room via the Spacer.

No code change needed. Confirmed rendering on device.

### Part 3: Climate sentence patterns ✅

**What was wrong:** The old `activityLabel` strings used middle-dot separators ("Eco · set to", "Idle · set to") that read oddly as sentences. The OFF state showed "System off in the Kitchen." — location phrase makes no sense when the system is off.

**New sentence table (user-facing copy, exact):**

| State | Rendered sentence |
|-------|-------------------|
| Heating (heat/eco mode) | Heating toward 70° in the Kitchen. |
| Cooling (cool/eco mode) | Cooling toward 68° in the Kitchen. |
| Idle with setpoint | Holding at 70° in the Kitchen. |
| Eco, heating | Eco mode, heating toward 70° in the Kitchen. |
| Eco, cooling | Eco mode, cooling toward 68° in the Kitchen. |
| Eco, idle | Eco mode, holding at 70° in the Kitchen. |
| System off | System is off. |

Key changes:
- "Idle · set to" → "Holding at" (reads naturally in a sentence)
- "Eco · set to" → "Eco mode, holding at" (comma separation, no middle dot)
- OFF: "System off in the Kitchen." → "System is off." (no location phrase)
- The OFF state now renders as a single `Text` without the target/location HStack

The location phrase "in the Kitchen" only appears when there is an active setpoint (mode ≠ .off).

### Part 4: +/- relocated into trinity card TARGET column ✅

**Where they moved:** TARGET column of the trinity card, inside the framed box. The value ("68°") and a vertical [+][−] stack sit on the same row:

```
TARGET
[68°]  [+]
       [−]
what we set
```

Vertical layout chosen over horizontal because:
- The trinity column is narrow (maxWidth: .infinity / 3 columns)
- Vertical matches the spatial logic: + is up (raise target), − is down (lower target)
- It mirrors real thermostat UI conventions
- Adding width for horizontal [−][68°][+] would crowd the trinity

Button size: 36pt circles (down from 56pt in the old bottom-row position). Smaller is correct — these are now embedded controls, not primary actions.

**Where they were removed from:** The bottom row of `ClimateSection` previously was:
```swift
HStack {
    [mode pills]   Spacer   [−][+]
}
```
Now it's just:
```swift
HStack {
    [mode pills]
}
```
The space vacated by the adjuster pair is absorbed by the mode pills row's existing layout — no visual gap, no speculative fill.

**OFF state:** When `climate.effectiveMode == .off`, trinity collapses to SYSTEM + HERE (no TARGET column). The +/− buttons disappear with the TARGET column. No separate hide logic needed — structural.

## Anything That Didn't Feel Right

The `targetCol` function for the TARGET column duplicates some structure from `trinityCol`. It's deliberate — TARGET needed custom content (the value + button pair) that doesn't fit the generic `(label, value, sub)` signature. But if trinity columns gain more variation in the future, the helper should be redesigned as a more flexible builder. Flagging as a future cleanup candidate.

The 36pt adjuster buttons inside the trinity card feel slightly small on the iPad Pro at 12.9". They're correct for touch target purposes (36pt within an 80pt+ touch area), but the visual affordance is subtle. If DC finds them too small after living with the panel for a day, 40pt circles would be the first adjustment to try.
